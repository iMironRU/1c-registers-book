# Гайд по стилю

> Claude Code и редакторы-сабагенты используют этот файл при написании и проверке
> параграфов. Главная аудитория — первокурсник; всё, что увеличивает его тревогу,
> мы выкидываем.

---

## Структура параграфа

Каждый параграф начинается с заголовка `#` и **строго одной темы**. Если в
процессе появляется вторая принципиально новая тема — это сигнал **разбивать на
два параграфа** (см. §«Одно новое за параграф» ниже).

Рекомендуемый объём: 1500–3000 слов. Короткий мостиковый параграф (как §A2) —
500–1000 слов.

Структура внутри:

1. **Зачин** — образ или вопрос, который ставит проблему.
2. **Содержательные разделы** — каждый закрывает одну подмысль.
3. **Итог** — короткая сводка (3–5 предложений), не повторяющая дорожную карту.
4. **Контрольные вопросы** — проверяют материал параграфа, не более 5.
5. **Упражнения** — лестница уровней (трассировка → копия → ловушка → чужая
   форма / чужой код).
6. **Ответы в трёх уровнях** — уровень 0 (на пальцах), уровень 1 (основной),
   уровень 2 (экспертный конспект). Плюс разбор каждого упражнения.
7. **Мостик к следующему параграфу** — формулирует ожидание, а не пересказывает
   тематику.

## Тон

- Прямое обращение к читателю — «ты». Прятки за «читатель», «пользователь»,
  «разработчик» не используем.
- Без канцелярита: «следует отметить», «представляется целесообразным»,
  «осуществляется», «производится» — нельзя.
- Активный залог, где возможно. Пассив — только при действительной
  необходимости.
- Без «по сути своей», «безусловно», «несомненно» — это пустые усиления.
- Без шаблонных зачинов «В данном параграфе мы рассмотрим…».
- Энергия — читатель должен чувствовать движение между предложениями. Где
  провисло — переписать.

---

## Никаких аббревиатур типа РС / РН / РБ / РР

Целевой читатель — первокурсник. Он только что познакомился с регистрами как
понятием. Каждая аббревиатура — это требование запомнить ещё одно сокращение.
Мы такого требования **не предъявляем**.

В прозе **всегда полное название**:

- «регистр сведений», «регистра сведений», «регистре сведений», «регистры
  сведений»…
- «регистр накопления», «регистры накопления»…
- «регистр бухгалтерии», «регистр расчёта».

**Исключения:**

- **Код 1С** — типы вроде `РегистрСведений.<Имя>`, `РегистрНакопления.<Имя>`.
  Это синтаксис платформы, его не меняем.
- **Прямые цитаты дерева конфигурации** — там как написано, так и пишем.
- **Таблицы сравнения**, где краткость колонок важна для читабельности. И там
  желательно полное название хотя бы в заголовке.

В черновиках, где уже встречаются «РС/РН/РБ/РР», **при следующей правке параграфа
расшифровываем**.

## Повторить вместо ссылки

Если читателю нужно вспомнить что-то из предыдущего параграфа, **повторяем
кратко прямо здесь**, а не отсылаем «см. §A3», «как было в §B1». Каждый параграф
должен стоять на своих ногах.

- ❌ «Помнишь, в §A3 мы говорили про скелет?»
- ✅ «У регистра есть три кости — измерения, ресурсы, реквизиты. Это его общий
   скелет; мы собрали его в концепте.»

Ссылки на параграфы допустимы как **дополнение** — для тех, кому хочется
подробнее: «полный разбор — там-то». Они **не заменяют** объяснение здесь.

Принцип: даже если читатель открыл параграф из середины книги и не помнит
предыдущие, он должен понимать суть. Ссылки — для углубления, не для опоры.

## Одно новое за параграф

Принцип «одно новое за раз» из CLAUDE.md §3.7 усиливаем:

- Если в параграфе появилось **второе принципиально новое понятие**, не
  вытекающее естественно из первого, — это **сигнал разбить параграф**.
- Лучше два коротких параграфа с одной мыслью каждый, чем один длинный со
  смешанными.
- «Расширение» уже введённого понятия второму новому понятию не считается:
  если ввели измерения, в этом же параграфе можно разворачивать «измерение как
  ось разреза», «измерение как ключ» — это всё про измерения.
- Если параграф приходится разбивать — нумерация подстраивается (или вводится
  «§B3.5»-стиль для дробных номеров).

## Перегрузка терминов — главный враг

Платформа 1С богата на слова, которые живут в нескольких ролях. Особенно
опасны:

- **«Остатки»** — имя регистра («ОстаткиТоваров»), вид регистра («регистр
  накопления с видом “Остатки”») и виртуальная таблица («`.Остатки`»). В одном
  абзаце все три значения — катастрофа.
- **«Период»** — свойство регистра сведений («Периодичность»), служебное поле
  записи («Период»), параметр виртуальной таблицы («Период»).
- **«Движение»** — строка в регистре и действие проведения документа.
- **«Запись»** — единица данных в регистре, действие глагола `Записать()`,
  процесс «разговор записи».

**Правила:**

1. Когда вводим слово в новой роли, **сразу проговариваем смысл**: «здесь
   «Остатки» — это **вид регистра** в конфигураторе, не виртуальная таблица».
2. В одном абзаце **не используем то же слово в двух разных ролях**, кроме
   крайних случаев. Если без этого никак — обозначаем явно.
3. Для имени регистра в сквозном примере по возможности выбираем имя, которое
   **не совпадает** с виртуалкой. Регистр накопления для учёта товаров —
   «**ТоварыНаСкладах**», не «ОстаткиТоваров». Иначе в коде вырастает
   «`РегистрНакопления.ОстаткиТоваров.Остатки`» с тройной перегрузкой.

## Метафоры и канон

См. CLAUDE.md §4 — там зафиксированы каноничные образы (фотоальбом, история
операций по карте, сеть переводов, расписание с правилами пересечения).
Запрещённые: «сообщающиеся сосуды», «откуда → куда = Кт → Дт» как мнемоника.

При выборе нового образа спросить:
- Выдерживает ли простой контрпример?
- Не конфликтует ли с другими образами в этом же параграфе?
- Можно ли его уплотнить до одного предложения, не теряя силу?

Метафора — несущая стенка объяснения, не декорация. Если её можно убрать без
потери смысла — лучше убрать.

## Примеры кода (с фазы B)

- Каждый пример **компилируется и работает** (как минимум синтаксически).
- Комментарии в коде на русском, объясняют «зачем», не «что».
- После каждой команды — точка с запятой. Закрытие блоков — `КонецЕсли`,
  `КонецЦикла`, `КонецПроцедуры`. Комментарии — только через `//`.
- Имена переменных на **русском** (как в типовой 1С-практике): `СтрокаТовара`,
  `НачалоПериода`, `ВыбраннаяНоменклатура`.
- Параметры запроса — через `&Имя`, никогда не подставлять через склейку строк.

## Тяжёлое форматирование

Книга — **проза**. Жирный, курсив, списки — точечно, как акценты, не как способ
организации мысли.

- Жирный — для **ключевых слов**, которые надо запомнить.
- Курсив — для названий, цитат, мягких акцентов.
- Списки — когда речь действительно идёт о наборе равноправных пунктов
  (правила, ошибки начинающих). Если можно прозой — лучше прозой.
- Эмодзи — никогда (если только сам автор явно не попросит).

Если в параграфе списки и жирный занимают больше четверти объёма — это
презентация, а не книга.

## Что не делать

- Не начинать параграф с «В данном разделе мы рассмотрим…».
- Не использовать пассивный залог там, где можно активный.
- Не ставить вопрос без ответа в том же параграфе (за исключением намеренного
  крючка — но тогда явно).
- Не оставлять «висячих» обещаний типа «об этом позже» без указания, где именно
  позже (если очень нужно — даём короткий намёк по сути, а ссылку даём
  дополнением).
- Не оставлять авторских заметок и метакомментариев («решения автора»,
  «возможно, тут стоит подумать») в основном тексте параграфа. Эти заметки —
  в `reviews/author-notes/<пара>.md` (локально).

## Сверка перед релизом

Фактические утверждения о платформе — особенно про **сигнатуры виртуальных
таблиц**, **имена служебных полей**, **значения по умолчанию свойств** — все
сверять с синтакс-помощником целевой версии 8.3.x перед релизом. См. список в
release-checklist (когда он будет создан) и в `reviews/*/synthesis.md` под
заголовком «На сверку перед релизом».
